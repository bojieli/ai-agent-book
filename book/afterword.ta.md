# பின்னுரை: Agent = LLM + Context + Tool எனும் அடிப்படைக்கு மீண்டும் {.unnumbered}

இந்தப் புத்தகம் ஒரு சூத்திரத்துடன் தொடங்கியது: **Agent = LLM + Context + Tool**. பத்து அத்தியாயங்களும் இந்த மூன்று வார்த்தைகளுக்குள் விரிகின்றன.

**அத்தியாயம் 1** சூத்திரத்தின் மூன்று-அடுக்கு புரிதலை நிறுவுகிறது—implementation layer, intuition layer, மற்றும் academic layer—மேலும் workflows முதல் autonomous agents வரையிலான orchestration-ன் ஒரு நிறமாலையை வழங்குகிறது. அத்தியாயங்கள் 2 முதல் 7 வரை பின்னர் சூத்திரத்தின் மூன்று தூண்களை ஒவ்வொன்றாக உருவாக்குகின்றன:

- **Context (அத்தியாயங்கள் 2 & 3)—"Agent பார்ப்பது என்ன."** Context engineering ஒரு ஒற்றை session-ல் model பார்க்கும் உலகத்தை தீர்மானிக்கிறது; user memory மற்றும் knowledge bases இதை ஒரு session-லிருந்து பல session-களுக்கு நீண்ட கால திரட்சியாக விரிவுபடுத்துகின்றன.
- **Tools (அத்தியாயங்கள் 4 & 5)—"Agent என்ன செய்ய முடியும்."** Tools திறனின் எல்லைகளை வரையறுக்கின்றன. Coding Agent-க்கு தனி அத்தியாயம் வழங்கப்பட்டுள்ளது, ஏனெனில் code மிகவும் உலகளாவிய tool—புதிய tools-ஐ உருவாக்கக்கூடிய ஒரு tool.
- **Model (அத்தியாயங்கள் 6 & 7)—"Agent-ன் மூளை தானே."** Evaluation agent-ன் செயல்திறனை ஒப்பிடக்கூடிய signals-ஆக மாற்றுகிறது; post-training பின்னர் இந்த signals-ஐ model-ன் சொந்த திறன்களாக மாற்றுகிறது. ஒன்று நுண்ணறிவை அளவிடுகிறது, மற்றொன்று அதை பெருக்குகிறது, மேலும் அவை ஒன்றாக சூத்திரத்தில் உள்ள LLM என்ற சொல் எவ்வளவு தூரம் செல்ல முடியும் என்பதை தீர்மானிக்கின்றன.

அத்தியாயங்கள் 8 முதல் 10 வரை பின்னர் இந்த மூன்று தூண்களையும் இணைத்து மிகவும் சிக்கலான சூழ்நிலைகளுக்கு பயன்படுத்துகின்றன:

- **அத்தியாயம் 8—Self-Evolution.** Agent தனது எடைகளை மாற்றாமல், அனுபவத்திலிருந்து உத்திகளை திரட்டவும், workflows-ஐ பதிவு செய்யவும், அறிவை வெளிப்படுத்தவும், மேலும் தீவிரமாக புதிய tools-ஐ உருவாக்கவும் உதவுகிறது.
- **அத்தியாயம் 9—Multimodal மற்றும் Real-Time Interaction.** உணர்வையும் செயலையும் text-லிருந்து vision, speech, மற்றும் physical world-க்கு விரிவுபடுத்துகிறது, real-time தேவைகளால் ஏற்படும் architectural சவால்களை எதிர்கொள்கிறது.
- **அத்தியாயம் 10—Multi-Agent Collaboration.** இது சூத்திரத்திற்கு வெளியே புதிதான ஒன்றல்ல: context பகிரப்படுகிறதா என்பது அத்தியாயம் 2-ன் "isolation over compression"-ஐ system architecture மட்டத்தில் வெளிப்படுத்துவதாகும்; "agents as tools for each other" என்பது அத்தியாயம் 4-ன் collaborative tool design-லிருந்து நேரடியாக வருகிறது; மேலும் multi-agent systems-ன் தரத்தை மதிப்பிடுவதற்கான "new information" அளவுகோல் அத்தியாயம் 6-ன் evaluation-ன் மையக் கருத்துக்களை எதிரொலிக்கிறது.

## இரண்டு மேகங்கள் {.unnumbered}

1900-ல், Lord Kelvin இயற்பியலின் தெளிவான வானத்தில் இரண்டு மேகங்கள் மிதப்பதாக கூறினார்—ஒன்று பின்னர் relativity ஆகவும், மற்றொன்று quantum mechanics ஆகவும் மாறியது. இன்று, agents-ன் மீதான வானமும் சமமாக தெளிவற்றுள்ளது, மேலும் நானும் இரண்டு மேகங்களைக் காண்கிறேன்.

**முதல் cloud என்பது agents எவ்வாறு தங்கள் environment உடன் streaming, real-time முறையில் தொடர்பு கொள்கின்றன என்பதாகும்.** இன்று, பெரும்பாலான agents இன்னும் turn-by-turn "request-response" முறையிலேயே செயல்படுகின்றன: நீங்கள் ஒரு வாக்கியத்தை முடிக்கிறீர்கள், அது ஒரு முழு பத்தியையும் சிந்தித்து, பின்னர் முடிவை ஒரே நேரத்தில் வெளியிடுகிறது. ஆனால் உண்மையான உலகம் அது சிந்தித்து முடிக்கும் வரை காத்திருக்க நிற்பதில்லை—பேச்சு குறுக்கிடப்படுகிறது, காட்சிகள் தொடர்ந்து மாறிக்கொண்டே இருக்கின்றன, மின்னஞ்சல்கள் வந்துகொண்டே இருக்கின்றன. உண்மையிலேயே "living" agent ஆனது சிந்திக்கும்போதே கேட்கவும், சிந்திக்கும்போதே பேசவும், நீங்கள் ஒரு வாக்கியத்தின் நடுவில் இருக்கும்போதே திட்டமிடத் தொடங்கவும், யாரும் கேட்காதபோதும் "இந்த மின்னஞ்சலை கவனிக்க வேண்டும்" என்பதை முன்கூட்டியே கவனிக்கவும் முடிய வேண்டும். இந்த real-time திறனை நோக்கி இரண்டு பாதைகள் உள்ளன, அவை பெரும்பாலும் இணையாகப் பின்பற்றப்படுகின்றன: ஒன்று **fast மற்றும் slow இன் architectural separation**—real-time மற்றும் intelligence என்பவை கிட்டத்தட்ட orthogonal axes ஆகும், ஒரு single model இரண்டையும் கையாள்வது கடினம், எனவே ஒரு fast front-end model உரையாடலின் rhythm ஐ பராமரிக்கும் அதே வேளையில் ஒரு slow back-end model ஆழமான சிந்தனையை கையாள்கிறது; மற்றொன்று **inference ஐ வேகமாக்குவது**—decode speed போதுமான அளவு அதிகமாக இருக்கும்போது, turn-by-turn க்கான காத்திருப்பு நேரம் மிகவும் குறுகியதாகி கிட்டத்தட்ட மறைந்துவிடும், இது turn-by-turn மற்றும் "real-time" இடையேயான கோட்டை மங்கலாக்குகிறது. இந்த பாதை chips மற்றும் inference engines ஆல் வேகமாக முன்னேற்றப்படுகிறது: Xiaomi MiMo ஒரு 1T-parameter model ஐ ஒரு single 8-GPU node[^mimo] இல் 1000 token/s க்கும் அதிகமாக தள்ளியுள்ளது, அதே நேரத்தில் முழு model ஐயும் ஒரு chip இல் hardcode செய்யும் dedicated solutions (Taalas HC1 போன்றவை) ஒரு 8-billion-parameter model ஐ சுமார் 17000 token/s க்கும், 100 milliseconds க்கும் குறைவான response time உடன் தள்ளுகின்றன[^taalas]. ஒரு model வினாடிக்கு ஆயிரக்கணக்கான வார்த்தைகளை வெளியிட முடிந்தால், "think then speak" மற்றும் "think while speaking" இடையேயான அனுபவ இடைவெளி சமன் செய்யப்படுகிறது.

**இரண்டாவது cloud என்பது agents எவ்வாறு மனிதர்களைப் போலவே, தங்கள் environment உடனான தொடர்புகளில் வெற்றிகள் மற்றும் தோல்விகளில் இருந்து தொடர்ந்து அனுபவத்தைக் குவிக்க முடியும் என்பதாகும்.** இன்றைய models ஒரு சிறந்த நினைவாற்றல் கொண்ட ஆனால் புதிய விஷயங்களைக் கற்றுக்கொள்ள முடியாத ஒரு மேதையைப் போன்றவை: அவை பயிற்சியின் போது மனித அறிவை முழுமையாக மனப்பாடம் செய்கின்றன, ஆனால் deployment க்குப் பிறகு அரிதாகவே வளர்கின்றன—ஒவ்வொரு பணிக்குப் பிறகும், அவை மிதித்த குழிகள் மற்றும் அவை கண்டுபிடித்த தந்திரங்கள் பெரும்பாலும் context உடன் சேர்த்து நிராகரிக்கப்படுகின்றன. இது ஒரு உண்மையான பிரச்சனையா என்பது இரண்டு எதிரெதிர் hypotheses ஐப் பொறுத்தது.

ஒன்று **"Small World Hypothesis"** (சிறிய உலகக் கருதுகோள்): போதுமான அளவு பெரிய model—சொல்லப்போனால், டிரில்லியன் கணக்கான parameters உடையது—ஏற்கனவே இயற்பியல் உலகத்தின் முக்கியமான பொது அறிவு அனைத்தையும் கொண்டுள்ளது; ஒருமுறை கற்றால் போதும். இந்தக் கருத்தை வைத்திருப்பவர்கள் (OpenAI மற்றும் Anthropic ஆராய்ச்சியாளர்கள் உட்பட) AI இன்று நிரலாக்கத்தில் மிகவும் வலுவாக இருப்பதற்குக் காரணம், code ஆனது models க்கு சிறப்பானது என்பதால் அல்ல, மாறாக நிரலாக்கம்தான் மனிதர்களுக்கு மிகவும் திறந்த களமாகும்: கற்க ஏராளமான open-source code கிடைக்கிறது; ஆனால், பெரும்பாலான தொழில்களுக்குப் பொதுத் தகவல் அல்லது தரவு இருப்பதில்லை. எனவே, frontier labs உண்மையில் செய்வது, பல்வேறு தொழில்களுடன் ஒவ்வொன்றாக ஒத்துழைத்து, அவற்றின் சொந்த தொழில்முறை திறன்களை ஒரே பெரிய model இல் "distill" செய்வதாகும். இந்தக் கண்ணோட்டத்தின்படி, தடையாக இருப்பது model இன் திறன் அல்லது கற்கும் திறன் அல்ல, மாறாக போதுமான தரவு இருக்கிறதா என்பதுதான்—தரவை உள்ளிடவும், ஒருமுறை train செய்யவும், பிரச்சினை தீர்க்கப்படும்.

ஆனால் **"Big World Hypothesis"** (பெரிய உலகக் கருதுகோள்) ஒரு "ஒருமுறை train செய்" அணுகுமுறையால் மறைக்க முடியாத ஒரு அடுக்கைச் சுட்டிக்காட்டுகிறது: ஒரு குறிப்பிட்ட பயனருக்கு அல்லது ஒரு குறிப்பிட்ட நிறுவனத்திற்கு உரித்தான அறிவு. உங்கள் நிறுவனத்தின் coding standards, உங்கள் குழுவின் PPT தயாரிப்பு ரசனை, ஒரு குறிப்பிட்ட வாடிக்கையாளரின் தனித்துவமான மனநிலை—இவை எந்த training corpus இலும் இல்லை, மேலும் அவை தொடர்ந்து மாறிக்கொண்டே இருக்கும். எண்ணற்ற குறிப்பிட்ட சூழ்நிலைகளால் ஆன இந்த "பெரிய உலகத்திற்கு" பொருந்த, model ஆனது deployment க்குப் பிறகு தொடர்ந்து கற்றுக்கொள்ள வேண்டியதுதான்; அதை factory இல் முழுமையாகத் தயாரித்து வைக்க முடியாது. இதுதான் Chapter 3 இன் memory மற்றும் Chapter 8 இன் self-evolution ஆகியவை ஆராய்வதன் திசை: அனுபவத்தை structured code ஆக எழுத வேண்டுமா, knowledge base இல் சேமிக்க வேண்டுமா, அல்லது model parameters இல் மீண்டும் distill செய்ய வேண்டுமா? மேலும் சென்றால், AI ஏற்கனவே self-evolve செய்யத் தொடங்கிவிட்டது—Anthropic தொடர்ந்து பேசும் "AI for AI" மற்றும் மேலும் மேலும் நிறுவனங்கள் "AI for Science" இல் பணியாற்றுவது அனைத்தும், agents அறிவியலின் எல்லைகளை ஆராய விடுவதைப் பற்றியதே; மேலும் எல்லைகளுக்கு முடிவே இல்லை, மனப்பாடம் செய்ய தயாரான நிலையான பதில்கள் அல்லது corpora எதுவும் இல்லை. Agents தங்கள் சொந்த பரிசோதனைகளின் வெற்றி மற்றும் தோல்விகளில் இருந்து சுயாட்சியுடன் கற்றுக்கொள்ள முடியும், எல்லாவற்றுக்கும் மனிதர்களிடம் திரும்பிச் செல்லாமல். எனவே, model இன் வலிமையான திறன் இறுதியில் மனப்பாடம் அல்ல, மாறாக கற்றல் மற்றும் தகவமைப்புதான்.

இந்த இரண்டு மேகங்களும் ஒரு single model upgrade ஆல் அகற்றப்படாது. அவை இறுதியில் எவ்வாறு சமாளிக்கப்படும் என்பதைப் புரிந்துகொள்ள, முதலில் ஒன்றைத் தெளிவாகப் பார்க்க வேண்டும்: models மற்றும் agents ஒருபோதும் upstream மற்றும் downstream ஆக இருந்ததில்லை; அவை ஒன்றாகவே உருவாகின்றன.

## Models மற்றும் Agents இன் Co-Evolution {.unnumbered}

அந்த harness-களில் உள்ள fallback logic-ன் அடுக்குகளைத் திரும்பிப் பார்க்கும்போது—multi-level context compression, ஆயிரக்கணக்கான தோல்விகளுக்குப் பின்னரே முறியும் retries, pessimistic "unsafe" default permission judgments—இந்த வெளித்தோற்றத்தில் அசிங்கமான "spaghetti code"-ன் ஒவ்வொரு துண்டும், model இன்னும் நிலைகுலைந்து நிற்கும் இடத்தைப் பதிவு செய்கிறது. அடுத்த தலைமுறை model-கள் இந்த கட்டுப்பாடுகளை உள்வாங்கிக் கொள்ளும்போது, அதற்கான code-ஐ நீக்க முடியும்; மேலும், model-கள் அவற்றை உள்வாங்க முடிவதற்குக் காரணம், agent-கள் ஏற்கனவே உண்மையான வணிகச் சூழல்களில் இந்தப் பாதைகளை வகுத்து, அவற்றை அடுத்த சுற்று பயிற்சிக்கான signals-ஆகச் சுருக்கியிருப்பதுதான். பயனர்கள் உண்மையான சவால்களை முன்வைக்கிறார்கள், application layer, model தற்காலிகமாகச் சிறப்பாகச் செய்ய முடியாதவற்றை ஈடுசெய்ய harness-களைப் பயன்படுத்துகிறது, மேலும் இந்த ஈடுசெய்தல்கள் model-ன் அடுத்த மறு செய்கைக்கான பயிற்சி signals-ஆக மாறுகின்றன—இது ஒரு self-reinforcing flywheel, மேலும் இந்த flywheel மூலம்தான் இரண்டு மேகங்களும் படிப்படியாக விலக்கப்படுகின்றன.

மேலும், மக்கள் இரு முனைகளையும் வைத்திருக்கும் இடத்தில்தான் flywheel வேகமாகச் சுழல்கிறது. Anthropic, Claude Code உடன் செய்வது, தனது சொந்த model-ஐயும் தனது சொந்த harness-ஐயும் ஒன்றையொன்று ஊட்டி, co-evolve செய்ய விடுவதுதான்: model-க்கு harness எப்படி அழைக்கும் என்பது தெரியும், harness-க்கு model-ன் எல்லைகள் தெரியும்; ஒவ்வொரு மாற்றமும் உடனடியாக மறுபக்கத்திற்குப் பின்னூட்டமாகச் செல்லும். ஒருவர் ஒரு பரிசோதனையில், model-ஐ மாற்றாமல், harness-ஐ மட்டும் மாற்றியதில், task accuracy 52.8% இலிருந்து 66.5% ஆகத் தாவியது—இது harness-க்கு இன்று எவ்வளவு leverage உள்ளது என்பதைக் காட்டுவதோடு, உங்களுக்கு ஒரு நினைவூட்டலையும் தருகிறது: model இன்னும் அந்த நிலையை அடையாததால்தான் இவ்வளவு leverage உள்ளது. இந்தக் காரணத்திற்காகவே, இந்த flywheel தானே இந்தக் காலகட்டத்தின் ஆழமான moat ஆகும்: உண்மையான வணிகம், feedback data, மற்றும் model iteration ஆகியவற்றுக்கு இடையேயான இணைப்பு எவ்வளவு இறுக்கமாக இருக்கிறதோ, அவ்வளவுக்கு மற்றவர்கள் வெளியில் இருந்து பிடிப்பது கடினம்.

இது உங்களுக்கு என்ன அர்த்தம் என்பது, நீங்கள் flywheel-ன் எந்த முனையில் நிற்கிறீர்கள் என்பதைப் பொறுத்தது. நீங்கள் model-களை உருவாக்குபவராக இருந்தால், moat என்பது இந்த flywheel-ஐச் சுழற்றுவதுதான்—உண்மையான உலக சூழல்களில் இருந்து வரும் feedback-ஐ முடிந்தவரை விரைவாகப் பயிற்சிக்குத் திருப்பி அனுப்புங்கள். நீங்கள் model-களின் மேல் applications-ஐ உருவாக்குபவராக இருந்தால், harness உங்களின் கூர்மையான குறுகியகால தொழில்நுட்ப அந்நியச் செலாவணியாகும், ஆனால் தெளிவாக இருங்கள்: model ஒவ்வொரு முறை ஒரு அடுக்கு கட்டுப்பாடுகளை உள்வாங்கும்போதும், harness-ஐ மட்டுமே நம்பி கட்டப்பட்ட ஒரு தொகுதி நன்மைகளை அது சாதாரணமாகத் தட்டையாக்கிவிடும். Application layer-ல் உண்மையான நீடித்த moat பெரும்பாலும் தொழில்நுட்பத்திற்கு வெளியே உள்ளது—தனிப்பட்ட data, நிலையான channels, பயனர் நம்பிக்கை, network effects, மற்றும் AI குறுகிய காலத்தில் கையாள முடியாத மற்றும் மனித-agent ஒத்துழைப்பு தேவைப்படும் அந்த சிக்கலான சூழ்நிலைகள். Harness-ஐப் பயன்படுத்தி நேரத்தை வாங்கி, அந்த நேரத்தைப் பயன்படுத்தி தொழில்நுட்பத்திற்கு வெளியே தடைகளைக் கட்டுவதே, விவேகமான அணுகுமுறையாகும்.

எனவே, நீங்கள் பயன்படுத்தும் framework வழக்கற்றுப் போய்விடும் என்று கவலைப்பட வேண்டாம். Models ஒவ்வொரு சில மாதங்களுக்கும் மீண்டும் செயல்படுகின்றன; குறிப்பிட்ட APIs, தயாரிப்புகள் மற்றும் leaderboards மாறிக்கொண்டே இருக்கும். ஆனால் மூன்று கேள்விகள்—"அது என்ன பார்க்கிறது, அது என்ன செய்ய முடியும், அது சரியாகச் செய்கிறதா என்பதை எவ்வாறு சரிபார்ப்பது"—வழக்கற்றுப் போகாது. அவை ஒரு குறிப்பிட்ட model-ன் பயன்பாட்டை அல்ல, மாறாக ஒரு அறிவார்ந்த அமைப்பு உலகத்துடன் தொடர்பு கொள்ளும் அடிப்படை வழியை விவரிக்கின்றன. அவற்றை மாஸ்டர் செய்யுங்கள், அடுத்த தலைமுறை models கொண்டு வரும் புதிய திறன்கள் எதுவாக இருந்தாலும், அதை formula-வில் எங்கு வைக்க வேண்டும் என்பதை நீங்கள் அறிவீர்கள், மேலும் அந்த இரண்டு மேகங்களையும் அகற்றுவதில் இருந்து அது இன்னும் எவ்வளவு தூரம் உள்ளது என்பதை ஒரு பார்வையில் பார்க்க முடியும்.

Agent தொழில்நுட்பம் இன்னும் வேகமாக வளர்ந்து வருகிறது; ஒரு புத்தகம் அனைத்து மாற்றங்களையும் தொடர்ந்து பிடிக்க முடியாது. ஆனால் இந்த புத்தகம் உங்களுக்கு சில API-யின் குறிப்பிட்ட பயன்பாட்டை அல்ல, மாறாக தொழில்நுட்ப அலைகளுக்கு மத்தியில் தெளிவான மனநிலையுடன் இருக்க ஒரு தொகுப்பு தீர்ப்பு திறன்களை வழங்கினால், அது அதன் நோக்கத்தை நிறைவேற்றியதாகும். இந்த புத்தகத்தில் உள்ள அனைத்து முக்கிய உரை, விளக்கப்படங்கள் மற்றும் உடன் வரும் பரிசோதனை code ஆகியவை open source ஆகும். நீங்கள் repository-க்குச் சென்று, பரிசோதனைகளை நீங்களே இயக்கி, issues மற்றும் PRs சமர்ப்பிக்க வரவேற்கப்படுகிறீர்கள்—புரிதலுக்கும் செயலுக்கும் இடையே ஒரு ஆறு உள்ளது, அதை உங்கள் சொந்த கைகளால் மட்டுமே கடக்க முடியும். மேலும் agents-ன் மிகவும் கவர்ச்சிகரமான விஷயம் என்னவென்றால், அவை code எழுதுவதன் மூலம் புதிய திறன்களை உருவாக்க முடியும், தங்களை மேம்படுத்திக் கொள்ளவும் முடியும். இவ்வளவு தூரம் படித்துவிட்டீர்கள், நீங்கள் ஏற்கனவே "உருவாக்கத்தின்" கொள்கைகளை வைத்திருக்கிறீர்கள். இப்போது, சென்று ஏதாவது உருவாக்குங்கள்.
